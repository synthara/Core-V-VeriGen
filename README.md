## Quickstart

This repository contains utilities and supporting tools for **automatic RISC-V UVM Reference Model generation and performance analysis**.
It is designed to operate within the **OpenHW CORE-V UVM** environment and targets the **CV32E20** RISC-V core, 
including support for custom ISA extensions and interrupts.

### `autogen/` — Model Generation

This folder contains the main logic responsible for building the RISC-V UVM Reference Model automatically.
It handles everything from reading configuration files and instruction definitions to assembling the final SystemVerilog output.
 

- **`main.py`**              — main entry point for the model generator, it collects all the input data 
                               and turns it into consistent, ready to run model code. This script is used in a plug-and-play manner by running our provided flow script ( scripts/run-vcs.py), which automatically calls it. Alternatively, you can integrate it into your custom flow, or execute it as a standalone tool. 

- **`config/`**              — folder with input configuration files:  
  - `model_config.json`         — file with model parameters and settings, like class name, instruction width, and chosen extensions supported 
  - `instr_impl.json`           — file that contains the high level implementation of each supported instruction by the reference model  
  - `opcode_priority.json`      — file that controls how opcodes are matched in the decoder.
                                  Some instructions look almost the same and differ only by a few don’t care bits (x). If the decoder checks a generic pattern first, it can accidentally match an instruction that should have been decoded as something else

### `perf_analyzer/` — Performance Evaluation Tools

Provides scripts and configurations for analyzing and benchmarking the generated model’s simulation performance. These files are **tightly integrated into our flow** and NOT intended to be run if you are using your own custom flow

- **`perf_analyzer.py`**            — Main performance analysis script. It is primarily used to compare simulation performance against Spike, evaluating how different software programs and test configurations affect simulation time and CPU utilization 

- **`perf_analyzer_config.yml`** — Configuration file listing the UVM tests to be executed (currently only SPIKE and basic autogenerated model) and the set of program tests to run (e.g., hello-world, dhrystone, etc.). It is used by the performance analyzer to compare the UVM model against Spike.
Note: future versions will include support for the cvxif model, integrating -cop and -dmv flags.  




---




### `utils/` — Shared Python Utilities

Reusable helper modules used by the generator and analyzer tools. They handle configuration parsing, formatting, and data transformation.

- **`config.py`**                — This module defines utilities that prepare all environment variables and directory paths required for setup and model generation.   Note: This file needs to be moved out of this directory and it's only needed if you're running run-vcs
 
- **`fmt.py`**                   — This module defines a set of command templates used to automate the full build and simulation flow for the RISC-V UVM environment, in particular each command is a formatted script that is filled with runtime parameters

### `uvmt/` — UVM Integration Tests

The files in this folder are add-ons and overrides used in combination with other components (e.g., CV-X-IF, dual-reference RVFI).  
They are **not** the default CORE-V UVM testbench files: the default implementations live in the OpenHW CORE-V environment.  
Use these files here **only when** you need the choosed extended configurations. For baseline runs, rely on the default testbench files provided elsewhere.


#### `Additional files for cvxif integration`

- **`uvmc_cvxif_scoreboard.sv`**            — A UVM scoreboard that compares **CVXIF** transactions from the RTL core against the reference model, it is used in "uvmt_cv32e20_uvm_model_cvxif.sv"   

- **`uvmt_cv32e20_dut_wrap.sv`**            — A SystemVerilog wrapper around the CV32E20 core with all default features plus **CVXIF**-CORE bridge, it is used in "uvmt_cv32e20_uvm_model_cvxif.sv"

- **`uvmt_cv32e20_tb.sv`**                  — A UVM tb_top with all default features, plus **CVXIF** instances, it is used alongside "uvmt_cv32e20_uvm_model_cvxif.sv"

- **`uvmt_cv32e20_uvm_model_pkg.flist`**    — Filelist defining the source files and include paths for these RISC-V UVM model sources into the build environment 
                                            
- **`uvmt_cv32e20_uvm_model_pkg.sv`**       — SV package for shared UVM model definitions


#### `UVM Model tests and modules`

##### - `uvmt_cv32e20_uvm_model_rvfi.sv` 
    
 This file contains a set of classes that are mandatory to run the autogenerated reference model **alone** or **alongside with SPIKE**

 - **`uvmt_cv32e20_model_test_c`**  — Defines a UVM test that overrides SPIKE  with the autogenerated UVM model

 - **`uvme_cv32e20_sb_dual_ref_c`**  — Extends the standard scoreboard to include an additional RVFI scoreboard instance, allowing simultaneous comparison of SPIKE and autogenerated UVM reference model 

 - **`uvme_cv32e20_env_dual_ref_c`**  — Defines an extended UVM environment capable of instantiating and connecting two reference models alongside a dual scoreboard for parallel RVFI verification

 - **`uvmt_cv32e20_model_test_dual_ref_c`**  — Implements a dual-reference UVM test configuration that runs both the autogenerated UVM model and Spike in parallel, enabling side-by-side functional comparison through the dual-reference environment and scoreboard

##### - `uvmt_cv32e20_uvm_model_cvxif.sv`  

  This file contains a set of classes that are mandatory to run the autogenerated reference model **with the cvxif** instead of the Spike-based model

 - **`uvme_cv32e20_sb_with_cvxif_c`** — Extends the standard scoreboard by adding a dedicated CVXIF scoreboard to compare CVXIF request/response streams from RTL and the reference model.

 - **`uvme_cv32e20_env_with_cvxif_c`** — Extended environment that instantiates the CVXIF agent, propagates its config, and overrides the base scoreboard with the CVXIF-enabled scoreboard. Wires CVXIF transactions from the RTL monitor and from the reference model into the scoreboard for side-by-side verification

 - **`uvmt_cv32e20_model_test_with_cvxif_c`**  — Defines a UVM test that overrides SPIKE with the autogenerated UVM model and is intended to run with the CVXIF-aware environment

## `scripts/run-vcs.py` — Running the model

This is the script you **must** run in the shell `if you want to use our digital flow`. It autogenerates all the files needed to the model to work, then builds and starts the simulation. It also checks whether the wrapper-repo Spike files are already compiled; if not, it runs `script/compileSpike.py`. By default the script will create a log directory in the wrapper repo, where the design is compiled and the simulation run.

---

### 1. Overview of the Flow

When you execute `run_vcs.py`, it performs the following phases automatically:

1. **Environment setup** – configures paths, toolchain, and defines.
2. **Opcode parsing** – generates `inst.sverilog` from the ISA definitions.
3. **Model autogeneration** – creates the SystemVerilog reference model class (`uvmc_rvfi_decoder_model.sv`).
4. **Firmware compilation** – compiles BSP (boot code, handlers, syscalls) and the selected test program.
5. **RTL and UVM compilation** – builds the VCS simulation environment.
6. **Simulation execution** – launches VCS, runs the test, and collects results.

All commands for these stages are defined in `utils/fmt.py` and automatically executed by `run_vcs.py`.

---


### 2. How to Use It

#### Basic Run
```bash
python3 scripts/run-vcs.py 
```

This runs a complete default build and simulation using:
- SPIKE as reference model,
- the “hello-world” program test,
- the `rv32imc_zicsr` architecture profile.

#### Run with autogenerated RVFI model
```bash
python3 scripts/run-vcs.py   -test uvmt_cv32e20_model_test_c   -program <_program_to_run_>   -march rv32imc_zicsr
```

This runs a complete build and simulation using:
- the autogenerated RVFI model,
- the "_program_to_run_" firmware test,
- and the `rv32imc_zicsr` architecture profile.


#### Run with CVXIF (coprocessor and data mover)
```bash
python3 scripts/run-vcs.py   -test uvmt_cv32e20_model_test_with_cvxif_c   -program <_program_with_cv_instr_>   -march rv32imc_zicsr_xcvalu_xcvsimd_xcvmac_xcvmem   -cop -dmv
```

When you use `-cop` and `-dmv`:
- It is **mandatory** that you use `uvmt_cv32e20_model_test_with_cvxif_c` as **-test** flag and `rv32imc_zicsr_xcvalu_xcvsimd_xcvmac_xcvmem` as **-march** flag
- Adds the coprocessor.
- Includes the coprocessor and data mover filelists.
- Enforces that the chosen test name contains `cvxif`.

This runs a complete build and simulation using:
- the autogenerated RVFI + CVXIF model,
- the "_program_with_cv_instr_" firmware test,
- and the `rv32imc_zicsr_xcvalu_xcvsimd_xcvmac_xcvmem` architecture profile.

#### Run with Dual Reference (Spike + Autogen)
```bash
python3 scripts/run-vcs.py   -test uvmt_cv32e20_model_test_dual_ref_c   -program <_program_to_run_>
```

This runs the testbench with both the autogenerated model and the Spike reference model for comparison.

### 3. Supported UVM Tests

| Test Name | Description |
|------------|-------------|
| `uvmt_cv32e20_firmware_test_c` | Uses Spike as reference model (default) |
| `uvmt_cv32e20_model_test_c` | Uses only the autogenerated RVFI model |
| `uvmt_cv32e20_model_test_dual_ref_c` | Runs Spike and the autogenerated model together |
| `uvmt_cv32e20_model_test_with_cvxif_c` | Uses the autogenerated RVFI model alongside the CVXIF model |

---

### 4. Supported March Values

`scripts/run-vcs.py` validates that the selected `-march` flag is allowed.  
Available architectures include:

```
rv32imc_zicsr                  (default)
rv32imc
rv32imc_zicsr_xcvalu_xcvsimd_xcvmac_xcvmem
```

These correspond to progressively extended instruction sets including custom CV extensions.

---

### 5. Common Command-Line Options

Use `python3 scripts/run-vcs.py -h` to see all options. The most relevant:

| Flag | Description |
|------|--------------|
| `-test` | Choose which UVM test to run (see list above). |
| `-program` | Select firmware to compile and run (e.g., `hello-world`, `fibonacci`, custom simple_cv_* tests). |
| `-march` | Architecture configuration (validated by the script). |
| `-cop` / `-dmv` | Include coprocessor and data mover. Required for CVXIF tests. |
| `-toolchain` | Path to the RISC-V GCC toolchain prefix |

---

### 6. Output Files

After a successful run, results are located in:

```
log/default/vcs_results/default/<program>/0/
```

| File | Description |
|-------|-------------|
| `vcs-<test>_<program>.log` | Full VCS simulation log. |
| `<program>.yaml` | UVM summary report. |
| `<program>.elf` | Compiled firmware image. |
| `<program>.hex` | Memory image loaded into simulation. |
| `<program>.itb` | Instruction trace binary (for waveform inspection). |

### 7. Example Workflow Summary

1. **Enter the project directory**
   ```bash
   cd core-v-verif
   ```
2. **Run the full flow**
   ```bash
   python3 scripts/run-vcs.py -test uvmt_cv32e20_model_test_c -program hello-world
   ```
3. **Check the generated model**
   ```
   core-v-verif/lib/uvm_components/uvmc_rvfi_reference_model/uvmc_rvfi_decoder_model.sv
   ```
4. **Inspect simulation logs**
   ```
   log/default/vcs_results/default/hello-world/0/
   ```
5. **Modify JSON or run different tests** to add or change instructions, then rerun the same command.

### 8. Key Advantages

- Fully automated generation — no manual SystemVerilog coding.
- Supports standard, arithmetic, compressed and CV custom extensions.
- UVM compliant and automatically integrated into the UVM environment.
- Synchronization with RTL via RVFI and optional CVXIF transactions.
